UnDelay is running a three-person team that just stood up an LLM assistant on Llama 3 at a 23 million passenger airport, landed in Microsoft for Startups, and is on the 2025 Forbes 30 Under 30 list. That is a lot of surface for a pre-seed team. Most of the work still ahead is not AI work. It is context architecture, process design, and the unglamorous middle layer between the radio transcripts and the airport ops dashboard.

Here is how we would think about UnDelay's stack specifically. Call it a 60/30/10 read.

60 / 30 / 10 · UnDelay stack read

60%
Traditional code. Ingesting radio audio, running transcription, structuring events, dispatching to the dashboard, handling the virtual taxi queue. Postgres and a few well-named services do this work better, cheaper, and with better guarantees than any LLM.
30%
Rule-based logic. When does a curb event escalate, when does a taxi slot open, when does an airline ops seat get paged. Rules, state machines, human-readable policies.
10%
Genuine AI. Where UnDelay GPT actually earns its keep: summarizing a messy situation for a ground ops lead, answering a natural-language question against the event log, suggesting a reroute when the usual playbook does not apply.

That ten percent is where the context architecture matters, and it is also where a small team burns weekends if the context is not organized.

My colleague Jake wrote a paper on exactly this problem. Interpretable Context Methodology, submitted to ACM TiiS, open source under MIT. It treats agent context as a layered filesystem, L0 identity through L4 working artifacts, with measurable interpretability gains. A 52-member practitioner community, a U-shaped edit pattern across real teams. The short version: a three-person team can keep an assistant coherent across four airports if the context is laid out this way, and cannot if it is not. Repo: github.com/RinDig/Interpretable-Context-Methodology-ICM-.
